## 分类原则

### 分层原则

瓶颈分类遵循以下原则：

1. **自底向上**: 从硬件层到应用层
2. **逐层隔离**: 每层独立诊断
3. **跨层关联**: 识别跨层瓶颈传播
4. **根因定位**: 找到最底层的瓶颈根因

### 严重性分级

| 级别 | 名称 | 性能影响 | 处理优先级 |
|-----|------|---------|-----------|
| **Critical** | 严重 | 性能下降 > 50% | P0 - 立即处理 |
| **Warning** | 警告 | 性能下降 20-50% | P1 - 优先处理 |
| **Notice** | 注意 | 性能下降 5-20% | P2 - 建议关注 |
| **Info** | 信息 | 性能下降 < 5% | P3 - 可选优化 |

## 7 层瓶颈框架详解

### Layer 1: Compute (计算层)

#### 定义
计算层关注处理器（CPU/NPU/GPU）的算力利用率和计算效率。

#### 关键指标

| 指标 | 数据来源 | 正常范围 | 瓶颈阈值 |
|-----|---------|---------|---------|
| CPU 利用率 | /proc/stat | 60-80% | < 30% 或 > 90% |
| NPU 利用率 | NPU 驱动 | 70-90% | < 50% |
| GPU 利用率 | nvidia-smi | 70-90% | < 50% |
| IPC (Instructions Per Cycle) | perf | > 1.0 | < 0.5 |
| 算术强度 | 性能计数器 | > 10 FLOP/Byte | < 5 FLOP/Byte |

#### 瓶颈模式

##### 1. CPU Starvation (CPU 饥饿)

**定义**: CPU 利用率过低，计算资源未被充分利用。

**识别特征**:
- CPU 利用率 < 30%
- 低上下文切换率
- 高 I/O 等待时间

**根因**:
- 线程数量不足
- I/O 阻塞导致线程等待
- 锁竞争导致线程阻塞

**优化建议**:
- 增加工作线程数量
- 使用异步 I/O 减少阻塞
- 优化锁的使用

---

##### 2. NPU Idle (NPU 空闲)

**定义**: NPU 利用率过低，AI 加速器未被充分利用。

**识别特征**:
- NPU 利用率 < 50%
- 低内存带宽使用
- 低 PCIe 带宽使用

**根因**:
- 数据加载慢，NPU 等待数据
- 模型过小，无法充分利用 NPU
- 批次大小过小

**优化建议**:
- 优化数据加载流水线
- 增加批次大小
- 使用模型并行增加计算量

---

##### 3. Compute Bound (计算受限)

**定义**: 计算密集型任务，算力不足成为瓶颈。

**识别特征**:
- CPU/NPU 利用率 > 90%
- 高 IPC (> 1.5)
- 低缓存缺失率

**根因**:
- 模型计算量大
- 算法复杂度高
- 硬件算力不足

**优化建议**:
- 优化算法减少计算量
- 使用更高效的算子
- 增加计算资源

---

### Layer 2: Memory (内存层)

#### 定义
内存层关注内存带宽、缓存效率和内存访问延迟。

#### 关键指标

| 指标 | 数据来源 | 正常范围 | 瓶颈阈值 |
|-----|---------|---------|---------|
| 内存带宽利用率 | PCM | 40-60% | > 80% |
| L1 缓存命中率 | perf | > 90% | < 80% |
| L2 缓存命中率 | perf | > 85% | < 70% |
| LLC 缓存命中率 | perf | > 80% | < 60% |
| 内存延迟 | PCM | < 100ns | > 150ns |

#### 瓶颈模式

##### 1. Memory Bandwidth Wall (内存带宽墙)

**定义**: 内存带宽饱和，成为性能瓶颈。

**识别特征**:
- 内存带宽利用率 > 80%
- 高 LLC 缺失率
- 低算术强度

**根因**:
- 数据访问模式不缓存友好
- 内存带宽不足
- 多个进程竞争内存带宽

**优化建议**:
- 优化数据访问模式
- 使用缓存友好的数据结构
- 减少内存访问次数

---

##### 2. Cache Miss Storm (缓存缺失风暴)

**定义**: 缓存缺失率过高，导致大量内存访问。

**识别特征**:
- L1 缓存命中率 < 80%
- LLC 缓存命中率 < 60%
- 高内存带宽使用

**根因**:
- 工作集超过缓存容量
- 数据访问模式随机
- 缓存污染

**优化建议**:
- 减少工作集大小
- 优化数据局部性
- 使用缓存分区

---

##### 3. HBM Bandwidth Limit (HBM 带宽限制)

**定义**: HBM (高带宽内存) 带宽成为瓶颈。

**识别特征**:
- HBM 带宽利用率 > 80%
- NPU 利用率低
- 高内存访问延迟

**根因**:
- 模型参数量大
- 激活值内存占用大
- HBM 带宽不足

**优化建议**:
- 使用模型压缩
- 减少激活值内存占用
- 使用梯度检查点

---

### Layer 3: Interconnect (互连层)

#### 定义
互连层关注节点内互连带宽，包括 NUMA、PCIe、NVLink、HCCS 等。

#### 关键指标

| 指标 | 数据来源 | 正常范围 | 瓶颈阈值 |
|-----|---------|---------|---------|
| NUMA 远端访问比例 | numa_maps | < 20% | > 30% |
| PCIe 带宽利用率 | PCIe 计数器 | < 70% | > 80% |
| NVLink 带宽利用率 | NVLink 计数器 | < 70% | > 80% |
| NUMA 距离 | sysfs | < 20 | > 30 |

#### 瓶颈模式

##### 1. Cross-NUMA Access (跨 NUMA 访问)

**定义**: 大量跨 NUMA 节点访问，导致高延迟。

**识别特征**:
- NUMA 远端访问比例 > 30%
- CPU-MEM 相似度 < 0.5
- 高内存访问延迟

**根因**:
- 数据和线程 NUMA 亲和性不匹配
- 内存分配策略不当
- 多 NUMA 节点数据共享

**优化建议**:
- 调整 NUMA 亲和性
- 使用 numactl 绑定内存
- 数据分区减少跨 NUMA 访问

---

##### 2. PCIe Bottleneck (PCIe 瓶颈)

**定义**: PCIe 带宽饱和，成为数据传输瓶颈。

**识别特征**:
- PCIe 带宽利用率 > 80%
- 高 PCIe 延迟
- Host-Device 数据传输慢

**根因**:
- Host-Device 数据传输量大
- PCIe 带宽不足
- PCIe 设备竞争

**优化优化**:
- 减少 Host-Device 数据传输
- 使用 GPUDirect RDMA
- 使用 PCIe 4.0/5.0

---

##### 3. NVLink Contention (NVLink 竞争)

**定义**: NVLink 带宽竞争，影响 GPU 间通信。

**识别特征**:
- NVLink 带宽利用率 > 80%
- GPU 间通信延迟高
- 多 GPU 通信竞争

**根因**:
- GPU 间通信量大
- NVLink 拓扑不合理
- 多任务竞争 NVLink

**优化建议**:
- 优化 GPU 间通信模式
- 使用 NVLink 感知的通信库
- 减少 GPU 间通信

---

### Layer 4: Network (网络层)

#### 定义
网络层关注节点间通信，包括 TCP/IP、RDMA、集合通信等。

#### 关键指标

| 指标 | 数据来源 | 正常范围 | 瓶颈阈值 |
|-----|---------|---------|---------|
| 网络带宽利用率 | 网络监控 | < 70% | > 80% |
| 网络延迟 | ping/ibv | < 10μs | > 50μs |
| 包重传率 | 网络统计 | < 0.1% | > 1% |
| RDMA 重传率 | RDMA 统计 | < 0.01% | > 0.1% |

#### 瓶颈模式

##### 1. Network Congestion (网络拥塞)

**定义**: 网络带宽饱和或拥塞，导致通信延迟高。

**识别特征**:
- 网络带宽利用率 > 80%
- 高网络延迟
- 高包重传率

**根因**:
- 网络带宽不足
- 网络拓扑不合理
- 多任务竞争网络

**优化建议**:
- 升级网络带宽
- 优化网络拓扑
- 使用流量控制

---

##### 2. RDMA Bottleneck (RDMA 瓶颈)

**定义**: RDMA 带宽或资源不足，影响高性能通信。

**识别特征**:
- RDMA 带宽利用率 > 80%
- 高 RDMA 重传率
- RDMA 资源耗尽

**根因**:
- RDMA 带宽不足
- RDMA 内存注册过多
- RDMA 工作队列满

**优化建议**:
- 增加 RDMA 带宽
- 优化 RDMA 内存注册
- 增加 RDMA 工作队列深度

---

##### 3. Collective Communication Bottleneck (集合通信瓶颈)

**定义**: 集合通信（All-Reduce、All-Gather 等）成为瓶颈。

**识别特征**:
- 集合通信时间长
- 高网络带宽使用
- 通信延迟高

**根因**:
- 集合通信算法不当
- 通信数据量大
- 网络拓扑不匹配

**优化建议**:
- 选择合适的集合通信算法
- 使用通信计算重叠
- 优化通信数据量

---

### Layer 5: Storage (存储层)

#### 定义
存储层关注文件系统、对象存储、检查点等存储相关瓶颈。

#### 关键指标

| 指标 | 数据来源 | 正常范围 | 瓶颈阈值 |
|-----|---------|---------|---------|
| IOPS | iostat | 根据设备 | > 设备上限 80% |
| 存储吞吐量 | iostat | < 设备上限 70% | > 设备上限 80% |
| I/O 延迟 | iostat | < 10ms | > 50ms |
| I/O 队列深度 | iostat | < 32 | > 128 |

#### 瓶颈模式

##### 1. I/O Bottleneck (I/O 瓶颈)

**定义**: 存储带宽或 IOPS 不足，成为数据加载瓶颈。

**识别特征**:
- I/O 带宽利用率 > 80%
- 高 I/O 延迟
- 高 I/O 等待时间

**根因**:
- 存储设备性能不足
- I/O 模式不合理
- 多任务竞争存储

**优化建议**:
- 升级存储设备
- 优化 I/O 模式
- 使用缓存减少 I/O

---

##### 2. Checkpoint Bottleneck (检查点瓶颈)

**定义**: 检查点写入慢，影响训练效率。

**识别特征**:
- 检查点写入时间长
- 高存储带宽使用
- 训练暂停时间长

**根因**:
- 检查点数据量大
- 存储写入慢
- 同步写入阻塞

**优化建议**:
- 减少检查点频率
- 使用异步检查点
- 压缩检查点数据

---

##### 3. Data Loading Bottleneck (数据加载瓶颈)

**定义**: 数据加载慢，导致训练等待数据。

**识别特征**:
- 数据加载时间长
- 低存储带宽使用
- DataLoader 线程空闲

**根因**:
- 数据预处理慢
- 存储读取慢
- DataLoader 配置不当

**优化建议**:
- 优化数据预处理
- 使用内存缓存
- 增加 DataLoader worker 数量

---

### Layer 6: Control Plane (控制平面层)

#### 定义
控制平面层关注调度、协调、同步等控制逻辑。

#### 关键指标

| 指标 | 数据来源 | 正常范围 | 瓶颈阈值 |
|-----|---------|---------|---------|
| 调度延迟 | 调度器统计 | < 10ms | > 100ms |
| 同步等待时间 | 应用统计 | < 10% 总时间 | > 30% 总时间 |
| 锁竞争时间 | perf | < 5% 总时间 | > 20% 总时间 |
| 负载均衡度 | 应用统计 | 差异 < 20% | 差异 > 50% |

#### 瓶颈模式

##### 1. Scheduling Bottleneck (调度瓶颈)

**定义**: 任务调度延迟高，影响系统响应。

**识别特征**:
- 调度延迟 > 100ms
- 任务队列积压
- 低 CPU 利用率

**根因**:
- 调度算法复杂
- 任务数量多
- 调度器竞争

**优化建议**:
- 优化调度算法
- 减少任务数量
- 使用分布式调度

---

##### 2. Synchronization Bottleneck (同步瓶颈)

**定义**: 同步等待时间长，影响并行效率。

**识别特征**:
- 同步等待时间 > 30% 总时间
- 高锁竞争
- 屏障等待时间长

**根因**:
- 锁粒度过大
- 同步点过多
- 负载不均衡

**优化建议**:
- 减少锁粒度
- 减少同步点
- 使用无锁数据结构

---

##### 3. Load Imbalance (负载不均衡)

**定义**: 任务分配不均，部分节点过载或空闲。

**识别特征**:
- 负载差异 > 50%
- 部分节点 CPU 高，部分低
- 任务完成时间差异大

**根因**:
- 任务分配算法不当
- 任务大小不一致
- 节点性能差异

**优化建议**:
- 使用动态负载均衡
- 细粒度任务划分
- 考虑节点性能差异

---

### Layer 7: Data Plane (数据流处理层)

#### 定义
数据流处理层关注数据加载、预处理、流水线等数据处理流程。

#### 关键指标

| 指标 | 数据来源 | 正常范围 | 瓶颈阈值 |
|-----|---------|---------|---------|
| 数据加载吞吐量 | 应用统计 | > 训练吞吐量 | < 训练吞吐量 |
| 预处理延迟 | 应用统计 | < 计算时间 10% | > 计算时间 30% |
| 流水线气泡时间 | 应用统计 | < 10% 总时间 | > 30% 总时间 |

#### 瓶颈模式

##### 1. Data Loading Bottleneck (数据加载瓶颈)

**定义**: 数据加载速度慢于训练速度，导致训练等待。

**识别特征**:
- 数据加载吞吐量 < 训练吞吐量
- DataLoader 线程利用率高
- 训练线程等待数据

**根因**:
- 存储读取慢
- 数据预处理慢
- DataLoader 配置不当

**优化建议**:
- 使用 SSD 或内存缓存
- 优化数据预处理
- 增加 DataLoader worker

---

##### 2. Preprocessing Bottleneck (预处理瓶颈)

**定义**: 数据预处理慢，成为流水线瓶颈。

**识别特征**:
- 预处理延迟 > 计算时间 30%
- 预处理 CPU 使用率高
- 流水线阻塞

**根因**:
- 预处理算法复杂
- 预处理未并行化
- 数据增强过多

**优化建议**:
- 简化预处理算法
- 并行化预处理
- 减少数据增强

---

##### 3. Pipeline Bubble (流水线气泡)

**定义**: 流水线各阶段执行时间不匹配，产生气泡。

**识别特征**:
- 流水线气泡时间 > 30% 总时间
- 各阶段执行时间差异大
- 流水线利用率低

**根因**:
- 各阶段负载不均衡
- 流水线深度不当
- 微批次大小不合理

**优化建议**:
- 平衡各阶段负载
- 调整流水线深度
- 优化微批次大小

---

## 瓶颈模式库

### 模式命名规范

```
<Layer>-<Pattern>-<Severity>
```

示例:
- `Compute-NPUIdle-Warning`
- `Memory-BandwidthWall-Critical`
- `Network-Congestion-Notice`

### 模式属性

每个瓶颈模式包含以下属性：

```python
class BottleneckPattern:
    name: str                    # 模式名称
    layer: str                   # 所属层次
    severity: str                # 严重性
    description: str             # 描述
    key_metrics: List[str]       # 关键指标
    identification_rules: List   # 识别规则
    root_causes: List[str]       # 根因
    recommendations: List[str]   # 优化建议
    related_patterns: List[str]  # 相关模式
```

## 识别特征矩阵

### 特征-瓶颈映射表

| 特征 | 可能的瓶颈 | 概率 |
|-----|-----------|------|
| CPU 利用率 < 30% | CPU Starvation | 高 |
| CPU 利用率 > 90% | Compute Bound | 高 |
| NPU 利用率 < 50% | NPU Idle | 高 |
| 内存带宽 > 80% | Memory Bandwidth Wall | 高 |
| LLC 缺失率 > 40% | Cache Miss Storm | 高 |
| NUMA 远端访问 > 30% | Cross-NUMA Access | 高 |
| 网络带宽 > 80% | Network Congestion | 高 |
| I/O 等待 > 20% | I/O Bottleneck | 中 |
| 锁等待 > 20% | Synchronization Bottleneck | 高 |
| 负载差异 > 50% | Load Imbalance | 高 |

### 多特征组合识别

```python
def identify_bottleneck_by_features(features):
    """
    基于多特征组合识别瓶颈
    """
    # 特征组合 -> 瓶颈模式
    if features["cpu_usage"] < 30 and features["io_wait"] > 20:
        return "I/O Bottleneck"
    
    if features["cpu_usage"] > 90 and features["llc_miss_rate"] < 10:
        return "Compute Bound"
    
    if features["cpu_usage"] > 90 and features["llc_miss_rate"] > 40:
        return "Memory Bandwidth Wall"
    
    if features["numa_remote_access"] > 30:
        return "Cross-NUMA Access"
    
    if features["network_bandwidth"] > 80:
        return "Network Congestion"
    
    return "Unknown"
```
